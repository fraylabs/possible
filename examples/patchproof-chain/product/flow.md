# PatchProof product flow

## Product job

PatchProof has one job:

> Turn evidence a developer already has into a portable receipt that shows which
> completion claims the supplied evidence passes, fails, skips, leaves unknown,
> or does not support.

The user is finished when they can inspect the receipt, copy or download its
Markdown and JSON forms, reload the page, and recover the same local draft.
Generating a receipt never means the code is correct, secure, production-ready,
or wanted by customers.

## Primary browser flow

The complete flow is one screen and one local browser session:

1. **Start.** Open an empty draft or restore the last valid locally saved draft.
   The interface states which occurred. It makes no network request.
2. **Load evidence.** Choose the repository-local demo fixture or enter a task,
   unified diff, command/check log, optional artifact notes, and explicit
   limitations. Loading a fixture replaces a non-empty draft only after an
   explicit confirmation.
3. **Inspect extraction.** PatchProof lists changed paths and command outcomes.
   Binary patches, empty diffs, ambiguous logs, skipped commands, and non-zero
   exits remain visibly distinct; later successful output cannot hide an earlier
   failure.
4. **Classify claims.** Add or confirm claims about tests, changed files, UI, or
   other work. Each claim names the evidence it relies on. The rules in
   `product/data-contract.md` calculate its status. The interface does not offer
   a way to force an unsupported claim to `passed`.
5. **Generate.** Generate an inspectable receipt. It contains the task, evidence
   summaries, claim-to-evidence ledger, status counts, warnings, and limitations.
   Editing any receipt input makes the preview stale and disables copy/download
   until the user regenerates it.
6. **Export.** Copy or download Markdown and download normalized JSON. Export
   actions use the currently generated revision only and report success or a
   specific recoverable failure.
7. **Recover.** Reload the browser. The last valid draft returns from browser
   storage, including raw input, claims, artifacts, and limitations. A generated
   receipt may be regenerated byte-for-byte from the recovered inputs.

The primary-flow check must exercise steps 1–7 at desktop and mobile viewport
widths. Responsive layout may change, but terminology, evidence, statuses, and
available actions must not.

## Material failure and recovery

JSON import is transactional:

1. Read the selected file as untrusted text into temporary memory.
2. Parse JSON and validate the full schema before changing application state or
   browser storage.
3. If parsing or validation fails, keep the current draft byte-for-byte
   unchanged and display: what failed, the supported schema (`schemaVersion: 1`),
   and the actions **Choose another file** and **Keep current draft**.
4. If validation passes, show a replacement summary. Replace the current draft
   only after the user chooses **Import and replace**.

Canceling, dismissing, or failing an import cannot erase the current draft. An
unsupported future schema version is a validation failure, not a best-effort
import. Imported strings remain text; PatchProof never renders imported HTML,
executes a command, opens a supplied URL, or resolves a supplied path.

Other recoverable failures follow the same rule:

- If browser storage is unavailable or full, keep the in-memory draft, show
  **Draft is not saved in this browser**, and offer **Retry saving** plus JSON
  download. Do not claim refresh recovery.
- If clipboard access fails, keep the generated receipt and offer download.
- If a download cannot start, keep the generated receipt and show which export
  failed. Do not mark it downloaded.

## Truthful evidence rules

- `passed`, `failed`, `skipped`, `unknown`, and `unsupported` are five distinct
  values. They are never rendered as a binary success/failure badge.
- `passed` describes supplied evidence only. It must be calculated from valid,
  referenced evidence; a user-entered assertion alone cannot produce it.
- `failed` means supplied evidence explicitly contradicts the claim or records a
  non-zero outcome.
- `skipped` means the evidence explicitly says the check was not run.
- `unknown` means relevant evidence exists but is stale, ambiguous, incomplete,
  or unreadable (including a binary patch whose content cannot be inspected).
- `unsupported` means the required evidence is absent. Absence is never
  `passed`, `skipped`, or silently omitted.
- A receipt preserves unrelated changed files and explicit limitations rather
  than hiding them to make the outcome look cleaner.

The precise calculation and precedence rules live in
`product/data-contract.md`.

## Phase A fixture gate

The local product is not complete until deterministic tests cover all 12 cases:

| Fixture | Required observable result |
|---|---|
| Passing evidence | Referenced test claim is `passed`; the recorded command and zero exit remain inspectable. |
| Hidden non-zero exit | The relevant check and claim are `failed`, even if later output looks successful. |
| Skipped checks | The check and dependent claim are `skipped`, not `passed`. |
| Claimed tests without a test command | The claim is `unsupported`. |
| Claimed file absent from diff | The claim is `unsupported`, and the absent path is named. |
| UI completion without visual evidence | The UI claim is `unsupported`. |
| Stale or ambiguous logs | The relevant check and claim are `unknown`. |
| Unrelated changes | Every changed path remains in the receipt and unrelated paths are warned about. |
| Binary files | The path is preserved; content-dependent claims are `unknown`. |
| Empty diff | The empty state is explicit and file-change claims are `unsupported`. |
| Malformed import | Import is rejected and the current draft and saved draft are unchanged. |
| Explicit limitations | Limitations survive preview, Markdown, JSON, reload, and re-export verbatim. |

For every valid fixture, repeated generation from the same normalized input must
produce byte-equivalent JSON. Across all fixtures, absent required evidence must
never become `passed`.

## Scope and inherited uncertainty

This stage tests a narrow local product, not its market. It preserves these
inherited hypotheses as hypotheses:

- Developers will value a lightweight manual receipt builder across agents.
- A browser-only local workflow will be easier than reviewing raw logs.
- Portable Markdown and JSON receipts will fit existing review habits.

Willingness to use or pay, preferred capture method, distribution channel, and
commercial model remain unknown. PatchProof performs no repository execution,
upload, analytics, account creation, deployment, outreach, or network access.
